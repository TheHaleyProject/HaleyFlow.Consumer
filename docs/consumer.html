HaleyFlow Consumer

An attribute-driven handler framework for processing engine events. Write a class, decorate methods — the consumer library handles polling, dispatch, step tracking, ACK retry, and the transactional outbox automatically.

MariaDB C# / .NET Attribute-Driven At-Least-Once Step Tracking
Overview

What is HaleyFlow Consumer?

✅ What the consumer does
  • Polls the engine for due transition and hook events
  • Dispatches events to the correct handler method by attribute
  • Tracks per-event sub-steps in inbox_step (idempotent)
  • ACKs the engine via a transactional outbox (survives crashes)
  • Pins a handler version per entity instance on first event
  • Sends heartbeats to keep the engine's liveness check passing
🚫 What you do not need to do
  • Call AckAsync yourself — just return AckOutcome
  • Poll the engine — the consumer loops do that
  • Handle retry backoff — the outbox loop retries failed ACKs
  • Register by definition ID — names are resolved from the engine at startup
Key difference from the raw engine API
When using the engine directly, you subscribe to engine.EventRaised and call engine.AckAsync(…) yourself. With the consumer library, you write a LifeCycleWrapper subclass, return AckOutcome from your handler, and the library handles everything else — polling, delivery tracking, outbox retry, and ACK.
Getting Started

Quick Start

  1. 1
    Decorate your class — apply [LifeCycleDefinition("my-workflow")] to a LifeCycleWrapper subclass.
  2. 2
    Add handler methods — decorate with [TransitionHandler(eventCode)] or [HookHandler("route")].
  3. 3
    Implement abstract methods — OnUnhandledTransitionAsync and OnUnhandledHookAsync for unregistered events.
  4. 4
    Register with DI — add services.AddTransient<MyHandler>().
  5. 5
    Build and start — create WorkFlowConsumerService, call RegisterAssembly, then StartAsync.
// 1. Handler class
[LifeCycleDefinition("loan-approval")]
public class LoanApprovalHandler : LifeCycleWrapper {
    private readonly IEmailService _email;
    public LoanApprovalHandler(IEmailService email) { _email = email; }

    [TransitionHandler(100)]  // event code 100 = "submit"
    public async Task<AckOutcome> OnSubmitAsync(ILifeCycleTransitionEvent evt, ConsumerContext ctx) {
        await _email.SendConfirmationAsync(evt.EntityId, ctx.CancellationToken);
        return AckOutcome.Processed;
    }

    protected override Task<AckOutcome> OnUnhandledTransitionAsync(
        ILifeCycleTransitionEvent evt, ConsumerContext ctx)
        => Task.FromResult(AckOutcome.Processed);  // or Retry/Failed as needed

    protected override Task<AckOutcome> OnUnhandledHookAsync(
        ILifeCycleHookEvent evt, ConsumerContext ctx)
        => Task.FromResult(AckOutcome.Processed);
}

// 2. DI registration
services.AddTransient<LoanApprovalHandler>();

// 3. Build + start consumer
var consumer = new WorkFlowConsumerService(
    feed:    inProcessFeed,
    dal:     consumerDal,
    sp:      serviceProvider,
    options: new ConsumerServiceOptions {
        ConsumerGuid = "my-app-consumer",
        EnvCode      = 1
    }
);
consumer.RegisterAssembly(Assembly.GetExecutingAssembly());
await consumer.StartAsync(cancellationToken);C#
Core Concepts

Architecture

Background Loops

WorkFlowConsumerService.StartAsync launches three independent background loops on the thread pool:

HeartbeatLoop
  • Calls engine.BeatConsumerAsync on a timer
  • Default interval: every 30 seconds
  • Keeps the consumer "alive" from the engine's perspective
  • If heartbeat goes stale, engine stops dispatching to this consumer
PollLoop
  • Calls engine.GetDueTransitionsAsync + GetDueHooksAsync
  • Dispatches each item via a semaphore-bounded concurrency pool
  • If no events: sleeps for PollInterval (default 5s)
  • If events found: polls again immediately
OutboxLoop
  • Retries ACKs that failed to reach the engine
  • Reads pending rows from outbox table where next_retry_at <= now
  • If empty: sleeps for OutboxInterval (default 15s)
  • Survives process crashes — outbox is durable
Startup sequence
  • Scan all loaded assemblies for [LifeCycleDefinition] classes
  • Query engine to resolve definition names → IDs
  • Register consumer with engine, receive assigned consumer ID
  • Start the three loops
Engine DB (ack_consumer)
    │
    ▼ PollLoop — GetDueTransitionsAsync / GetDueHooksAsync
WorkFlowConsumerService
    │
    ├─ Semaphore (MaxConcurrency) ──► ProcessItemAsync
    │       ├─ Upsert workflow row    (consumer DB: workflow)
    │       ├─ Pin handler version    (first event only)
    │       ├─ Upsert inbox record    (consumer DB: inbox)
    │       ├─ LifeCycleWrapper.Dispatch ──► [TransitionHandler] / [HookHandler] method
    │       │       └─ Returns AckOutcome
    │       └─ Write outbox row ──► OutboxLoop ──► engine.AckAsync
    │
    └─ HeartbeatLoop ──► engine.BeatConsumerAsync

LifeCycleWrapper — The Handler Base Class

All workflow handlers inherit from LifeCycleWrapper. This base class provides step tracking helpers and the internal dispatch machinery. Subclasses are resolved from DI per event dispatch — constructor-inject your dependencies normally.

Step tracking methods (protected)

MethodPurpose
StartStepAsync(ctx, stepCode)Marks step as Running (2) in inbox_step; records started_at
CompleteStepAsync(ctx, stepCode, result?)Marks step as Completed (3); records completed_at and optional result JSON
FailStepAsync(ctx, stepCode, error?)Marks step as Failed (4); records completed_at and optional error message
IsStepCompletedAsync(ctx, stepCode)Returns true if the step is already in Completed state — use to skip idempotently

Required abstract methods

Implement these to handle events that have no matching [TransitionHandler] or [HookHandler] attribute:

protected abstract Task<AckOutcome> OnUnhandledTransitionAsync(
    ILifeCycleTransitionEvent evt, ConsumerContext ctx);

protected abstract Task<AckOutcome> OnUnhandledHookAsync(
    ILifeCycleHookEvent evt, ConsumerContext ctx);C#
Do not return Processed by default blindly
If you return Processed for all unhandled events, silently-ignored events will never re-deliver. Consider logging an alert and returning Failed for truly unexpected events, or Processed only if your handler deliberately handles all events.

Attributes

[LifeCycleDefinition("name")] — Class-level

Maps a LifeCycleWrapper subclass to a specific workflow definition by name. The name must match the defName field in the engine's definition JSON exactly (case-insensitive lookup).

[LifeCycleDefinition("loan-approval")]
public class LoanApprovalHandler : LifeCycleWrapper { ... }C#

[TransitionHandler(eventCode, minVersion?)] — Method-level

Marks a method as the handler for a specific transition event code. The method signature must be:

Task<AckOutcome> MethodName(ILifeCycleTransitionEvent evt, ConsumerContext ctx)C#
ParameterDescription
eventCodeThe integer event code from the engine's definition JSON (e.g. 100 for "submit")
minVersionMinimum handler version for this method to be selected. Default 0 = matches any version. See Handler Versioning section.

Marks a method as the handler for a hook with the given route name. The route must match the "event" field in the policy emit entry.

WrapperRegistry — Auto-Discovery

The registry maps definition IDs (engine-assigned) to handler types. It works in two phases:

  1. 1
    Assembly scan (RegisterAssembly) — finds all non-abstract types inheriting LifeCycleWrapper with [LifeCycleDefinition]. Stores them by definition name.
  2. 2
    Engine resolution (on StartAsync) — for each discovered name, calls engine.GetDefinitionIdAsync. Binds the returned def_id to the handler type.

At dispatch time, TryGetRegistration(defId) looks up the handler type. If no registration exists for an incoming definition ID, the event is silently skipped.

Multiple assemblies
Call RegisterAssembly for each assembly that contains handler classes before StartAsync. StartAsync also auto-scans all currently loaded assemblies as a fallback. 
consumer
    .RegisterAssembly(Assembly.GetExecutingAssembly())
    .RegisterAssembly("OtherHandlers.Assembly");  // name or already-loaded assembly
await consumer.StartAsync(ct);C#

Dispatch Flow (ProcessItemAsync)

When an event item is dequeued from the engine, ProcessItemAsync runs the following steps:

  1. 1
    Upsert workflow row — inserts a row in workflow table keyed by UNIQUE(consumer_id, ack_guid). Idempotent. Returns (wfId, isNew).
  2. 2
    Pin handler version (first event only) — if isNew, looks up the most recent handler version already pinned for this definition+entity. Pins it permanently on the workflow row.
  3. 3
    Upsert inbox — stores payload + params JSON in inbox table. Sets status to Processing, increments attempt count.
  4. 4
    Build context — creates ConsumerContext with WfId, AckGuid, HandlerVersion, CancellationToken, etc.
  5. 5
    Resolve handler from DI — resolves the wrapper type from IServiceProvider. Falls back to Activator.CreateInstance if not registered. Injects the step DAL.
  6. 6
    Dispatch — calls wrapper.DispatchTransitionAsync or wrapper.DispatchHookAsync. The dispatcher selects the best-matching handler method by MinVersion. Returns AckOutcome.
  7. 7
    Write outbox + attempt ACK — writes outcome to outbox table. Attempts to call engine.AckAsync immediately. If ACK fails, sets next_retry_at and the OutboxLoop retries later.
AckRequired = false
If the engine dispatched the event with no ack_guid (AckRequired was false), the consumer marks the outbox row as Confirmed immediately without calling the engine. No UNIQUE(consumer_id, ack_guid) deduplication occurs for these events — each arrives as a separate workflow row.

Step Tracking (inbox_step)

Each event dispatch can be broken into named steps stored in the inbox_step table. The primary key is (inbox_id, step_code) — so each step is idempotent across re-deliveries.

Why use steps?
  • At-least-once delivery means your handler may run multiple times
  • Steps let you skip already-completed work on re-delivery
  • Each step tracks started_at, completed_at, result, and error
  • Step codes are arbitrary integers — define them as constants in your handler
Step status values
  • 1 — Pending (initial)
  • 2 — Running (StartStepAsync)
  • 3 — Completed (CompleteStepAsync)
  • 4 — Failed (FailStepAsync)
// Idempotent step pattern — safe to run multiple times on re-delivery
const int STEP_VALIDATE = 1;
const int STEP_NOTIFY   = 2;

if (!await IsStepCompletedAsync(ctx, STEP_VALIDATE)) {
    await StartStepAsync(ctx, STEP_VALIDATE);
    try {
        await ValidateAsync(evt.EntityId);
        await CompleteStepAsync(ctx, STEP_VALIDATE, "ok");
    } catch (Exception ex) {
        await FailStepAsync(ctx, STEP_VALIDATE, ex.Message);
        return AckOutcome.Retry;  // engine will re-deliver
    }
}C#

Handler Versioning

Each entity instance has a handler version pinned on first event. This lets you deploy new handler logic without breaking in-flight instances running on older code.

How pinning works

  1. First event for an entity arrives. The service checks if any prior entity in this definition already has a pinned version (for consistency across a batch).
  2. The handler version is stored in the workflow row and never changes for that entity (unless HandlerUpgrade = AllowUpgrade).
  3. On every subsequent event, the pinned version from the workflow row is loaded into ctx.HandlerVersion.

MinVersion picker algorithm

For an event, the dispatcher collects all methods decorated with a matching attribute and picks the one with the highest MinVersion that is ≤ ctx.HandlerVersion.

// Example: three versions of the same event handler
[TransitionHandler(100, minVersion: 0)]   // selected when HandlerVersion = 0 or 1
public Task<AckOutcome> OnSubmitV1Async(...) { ... }

[TransitionHandler(100, minVersion: 2)]   // selected when HandlerVersion = 2 or 3
public Task<AckOutcome> OnSubmitV2Async(...) { ... }

[TransitionHandler(100, minVersion: 4)]   // selected when HandlerVersion >= 4
public Task<AckOutcome> OnSubmitV4Async(...) { ... }C#

HandlerUpgrade policy

ValueBehaviour
Pinned (default)Instance always uses its originally pinned version. Deploying new handler versions does not affect existing instances.
AllowUpgradeIf the wrapper now declares a higher MaxDeclaredVersion, the effective version is bumped to max(pinned, MaxDeclaredVersion). Existing instances pick up new handler methods on next delivery.
 
// Set upgrade policy in options (applies to new instances):
new ConsumerServiceOptions {
    DefaultHandlerUpgrade = HandlerUpgrade.Pinned  // default
}C#

Outbox & Reliability

The consumer uses a transactional outbox to guarantee ACK delivery even if the process crashes between completing work and sending the ACK to the engine.

Happy path
  • Handler runs, returns AckOutcome
  • Consumer writes outbox row (Pending)
  • Consumer calls engine.AckAsync immediately
  • On success: outbox row → Confirmed
ACK failure path
  • Handler completes but engine.AckAsync fails (network, timeout)
  • Outbox row stays Pending with next_retry_at
  • OutboxLoop retries on schedule (OutboxRetryDelay)
  • Each attempt is recorded in outbox_history
Crash safety
If the process crashes after the handler completes but before ACK is sent, the outbox row is durable in the consumer DB. When the process restarts, the OutboxLoop will pick up the pending row and ACK the engine. The engine will not re-deliver the event to this consumer as long as the ACK arrives within the engine's retry window.

AckOutcome — Return Values

Your handler returns one of these values. The consumer service translates this into an ACK call to the engine.

ValueEngine effectWhen to use
DeliveredACK status → DeliveredYou received the event but haven't finished processing yet. Use for two-phase ACK patterns.
ProcessedACK status → Processed (terminal)Work is fully done. No more re-delivery for this consumer.
RetryACK status → Pending + rescheduleTransient fault (network error, service down). Engine will re-deliver after AckPendingResendAfter.
FailedACK status → Failed (terminal)Permanent failure — you cannot process this event. If the hook was blocking, the engine may suspend the instance.
Retry count matters
If your handler returns Retry too many times, the engine's monitor will exceed MaxRetryCount and either suspend the instance (blocking hook) or fire ACK_FAIL (non-blocking hook). Investigate the root cause of transient failures rather than retrying indefinitely.

Consumer Database Schema

The consumer maintains its own MariaDB database (separate from the engine DB). It tracks the local processing state of each dispatched event.

TablePurposeKey Columns
workflow One row per dispatched event. The consumer's local record of each engine event. id, ack_guid, entity_id, kind (1=Transition, 2=Hook), consumer_id, def_id, handler_version, handler_upgrade, on_success, on_failure
inbox Stores payload + params for the current dispatch. Tracks processing status and attempt count. wf_id FK→workflow, payload_json, params_json, status (1=Received, 2=Processing, 3=Processed, 4=Failed), attempt_count, last_error
inbox_step Fine-grained step tracking within a single dispatch. PK = (inbox_id, step_code). inbox_id FK→inbox, step_code, status (1=Pending, 2=Running, 3=Completed, 4=Failed), started_at, completed_at, result_json, last_error
outbox Transactional outbox — one row per workflow. Tracks the outcome waiting to be ACK'd to the engine. wf_id FK→workflow, current_outcome (AckOutcome), status (1=Pending, 3=Confirmed), next_retry_at, last_error
outbox_history Audit log of every ACK attempt. Each row is one attempt with outcome, status, and optional response/error. outbox_id FK→outbox, outcome, status, attempt_no (auto-computed), response_payload_json, error

Example 2 — Transition Handler with Policy Params

Using OnSuccessEvent and OnFailureEvent from the policy to automatically trigger the next macro transition.

[LifeCycleDefinition("loan-approval")]
public class LoanApprovalHandler : LifeCycleWrapper {
    private readonly IWorkFlowEngine _engine;
    private readonly IReviewService   _review;

    public LoanApprovalHandler(IWorkFlowEngine engine, IReviewService review) {
        _engine = engine; _review = review;
    }

    [TransitionHandler(100)]  // submit → UnderReview
    public async Task<AckOutcome> OnSubmitAsync(
        ILifeCycleTransitionEvent evt, ConsumerContext ctx) {

        // Read policy params (from policy JSON "params" block)
        var slaHours = evt.Params?["sla_hours"]?.ToString() ?? "48";

        // Perform review work
        var approved = await _review.RunAsync(evt.EntityId, slaHours, ctx.CancellationToken);

        // Use OnSuccessEvent / OnFailureEvent from policy to know the next event code
        var nextEvent = approved ? evt.OnSuccessEvent : evt.OnFailureEvent;
        if (!string.IsNullOrWhiteSpace(nextEvent)) {
            await _engine.TriggerAsync(new LifeCycleTriggerRequest {
                EnvCode     = ctx.ConsumerId, // use your env code
                DefName     = "loan-approval",
                ExternalRef = evt.EntityId,
                Event       = nextEvent,
                RequestId   = $"auto-{evt.EntityId}-{nextEvent}",
                AckRequired = true
            }, ctx.CancellationToken);
        }

        return AckOutcome.Processed;
    }

    protected override Task<AckOutcome> OnUnhandledTransitionAsync(
        ILifeCycleTransitionEvent evt, ConsumerContext ctx)
        => Task.FromResult(AckOutcome.Processed);
    protected override Task<AckOutcome> OnUnhandledHookAsync(
        ILifeCycleHookEvent evt, ConsumerContext ctx)
        => Task.FromResult(AckOutcome.Processed);
}C#

Example 3 — Multi-Step Handler (Idempotent)

Break work into numbered steps. On re-delivery, completed steps are skipped automatically via IsStepCompletedAsync.

// Step code constants — define as an enum or const int
private const int STEP_VALIDATE = 1;
private const int STEP_RESERVE  = 2;
private const int STEP_NOTIFY   = 3;

[TransitionHandler(100)]
public async Task<AckOutcome> OnSubmitAsync(
    ILifeCycleTransitionEvent evt, ConsumerContext ctx) {

    // ── Step 1: Validate ──────────────────────────────────────
    if (!await IsStepCompletedAsync(ctx, STEP_VALIDATE)) {
        await StartStepAsync(ctx, STEP_VALIDATE);
        try {
            await _validator.CheckAsync(evt.EntityId, ctx.CancellationToken);
            await CompleteStepAsync(ctx, STEP_VALIDATE, "valid");
        } catch (ValidationException ex) {
            await FailStepAsync(ctx, STEP_VALIDATE, ex.Message);
            return AckOutcome.Failed;  // permanent — bad data, no retry
        } catch (Exception ex) {
            await FailStepAsync(ctx, STEP_VALIDATE, ex.Message);
            return AckOutcome.Retry;   // transient — retry later
        }
    }

    // ── Step 2: Reserve inventory ─────────────────────────────
    if (!await IsStepCompletedAsync(ctx, STEP_RESERVE)) {
        await StartStepAsync(ctx, STEP_RESERVE);
        try {
            var reservationId = await _inventory.ReserveAsync(evt.EntityId, ctx.CancellationToken);
            await CompleteStepAsync(ctx, STEP_RESERVE, reservationId);
        } catch (Exception ex) {
            await FailStepAsync(ctx, STEP_RESERVE, ex.Message);
            return AckOutcome.Retry;
        }
    }

    // ── Step 3: Notify applicant ──────────────────────────────
    if (!await IsStepCompletedAsync(ctx, STEP_NOTIFY)) {
        await StartStepAsync(ctx, STEP_NOTIFY);
        try {
            await _email.SendSubmittedAsync(evt.EntityId, ctx.CancellationToken);
            await CompleteStepAsync(ctx, STEP_NOTIFY);
        } catch (Exception ex) {
            // Non-critical — log but don't fail the whole event
            await FailStepAsync(ctx, STEP_NOTIFY, ex.Message);
        }
    }

    return AckOutcome.Processed;
}C#
On re-delivery
If the process crashes after Step 2 completes but before Step 3, the engine will re-deliver the event. Steps 1 and 2 are skipped (already Completed in inbox_step). Only Step 3 executes again. This makes multi-step handlers safe to re-run.

Example 4 — Hook Handler

Handle hook events emitted by the engine's policy. The route name in [HookHandler] must match the "event" field in the policy emit entry exactly.

[LifeCycleDefinition("loan-approval")]
public class LoanHookHandler : LifeCycleWrapper {
    private readonly IEmailService  _email;
    private readonly ICreditService _credit;

    public LoanHookHandler(IEmailService email, ICreditService credit) {
        _email = email; _credit = credit;
    }

    // Handles: "emit": [{ "event": "app.loan.notify_reviewer", ... }]
    [HookHandler("app.loan.notify_reviewer")]
    public async Task<AckOutcome> OnNotifyReviewerAsync(
        ILifeCycleHookEvent evt, ConsumerContext ctx) {
        // Params come from the policy "params" block
        var email = evt.Params?["reviewer_email"]?.ToString();
        if (string.IsNullOrWhiteSpace(email)) return AckOutcome.Failed;

        await _email.SendReviewRequestAsync(email, evt.EntityId, ctx.CancellationToken);
        return AckOutcome.Processed;
    }

    // Handles: "emit": [{ "event": "app.loan.run_credit_check", "blocking": true }]
    [HookHandler("app.loan.run_credit_check")]
    public async Task<AckOutcome> OnCreditCheckAsync(
        ILifeCycleHookEvent evt, ConsumerContext ctx) {
        try {
            var passed = await _credit.CheckAsync(evt.EntityId, ctx.CancellationToken);
            // Return Retry if transient; Processed/Failed for permanent outcomes.
            // If Failed on a blocking hook, engine may suspend the instance.
            return passed ? AckOutcome.Processed : AckOutcome.Failed;
        } catch {
            return AckOutcome.Retry;
        }
    }

    // Non-blocking audit log hook — failure is safe
    [HookHandler("app.loan.audit_log")]
    public Task<AckOutcome> OnAuditLogAsync(
        ILifeCycleHookEvent evt, ConsumerContext ctx) {
        _logger.LogInformation("Loan {id} entered review state", evt.EntityId);
        return Task.FromResult(AckOutcome.Processed);
    }

    protected override Task<AckOutcome> OnUnhandledTransitionAsync(
        ILifeCycleTransitionEvent evt, ConsumerContext ctx)
        => Task.FromResult(AckOutcome.Processed);
    protected override Task<AckOutcome> OnUnhandledHookAsync(
        ILifeCycleHookEvent evt, ConsumerContext ctx)
        => Task.FromResult(AckOutcome.Processed);
}C#

Example 5 — Handler Versioning

Deploy new logic for the same event code while keeping in-flight entities on their pinned version. Old entities continue using OnSubmitV1Async; new entities (with higher pinned versions) use the newer methods.

[LifeCycleDefinition("loan-approval")]
public class LoanApprovalHandler : LifeCycleWrapper {

    // ── Dispatch rules for event code 100 ──────────────────────────
    // HandlerVersion 0-1: use V1 (legacy flow)
    // HandlerVersion 2-3: use V2 (enhanced validation added)
    // HandlerVersion 4+ : use V4 (new credit pre-check)
    // The dispatcher picks the highest MinVersion that is ≤ HandlerVersion.

    [TransitionHandler(100, minVersion: 0)]
    public async Task<AckOutcome> OnSubmitV1Async(
        ILifeCycleTransitionEvent evt, ConsumerContext ctx) {
        // Original logic — existing in-flight entities still run this
        await _validator.BasicCheckAsync(evt.EntityId);
        return AckOutcome.Processed;
    }

    [TransitionHandler(100, minVersion: 2)]
    public async Task<AckOutcome> OnSubmitV2Async(
        ILifeCycleTransitionEvent evt, ConsumerContext ctx) {
        // New entities (HandlerVersion >= 2) run enhanced validation
        await _validator.EnhancedCheckAsync(evt.EntityId);
        await _notifier.SendReceiptAsync(evt.EntityId);
        return AckOutcome.Processed;
    }

    [TransitionHandler(100, minVersion: 4)]
    public async Task<AckOutcome> OnSubmitV4Async(
        ILifeCycleTransitionEvent evt, ConsumerContext ctx) {
        // Entities created after version 4 also run the credit pre-check
        await _validator.EnhancedCheckAsync(evt.EntityId);
        await _credit.PreCheckAsync(evt.EntityId);
        await _notifier.SendReceiptAsync(evt.EntityId);
        return AckOutcome.Processed;
    }

    // ── AllowUpgrade: let existing entities pick up new logic ──────
    // Set in ConsumerServiceOptions.DefaultHandlerUpgrade = AllowUpgrade
    // Entity with HandlerVersion=1 + AllowUpgrade will use V4 (max declared).
    // Only do this when the new logic is backwards-compatible.

    protected override Task<AckOutcome> OnUnhandledTransitionAsync(
        ILifeCycleTransitionEvent evt, ConsumerContext ctx)
        => Task.FromResult(AckOutcome.Processed);
    protected override Task<AckOutcome> OnUnhandledHookAsync(
        ILifeCycleHookEvent evt, ConsumerContext ctx)
        => Task.FromResult(AckOutcome.Processed);

Version pinning in practice
On first event for a new entity, the consumer looks up the highest handler version already in use for the same definition+entity. This means all events for one entity are processed consistently even when new handler versions are deployed mid-flight.
