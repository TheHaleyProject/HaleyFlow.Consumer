HaleyFlow Consumer

An attribute-driven handler framework for processing engine events. Write a class, decorate methods — the consumer library handles polling, dispatch, business action tracking, ACK retry, and the transactional outbox automatically.

MariaDB C# / .NET Attribute-Driven At-Least-Once Business Actions
Overview

What is HaleyFlow Consumer?

✅ What the consumer does
  • Polls the engine for due transition and hook events
  • Dispatches events to the correct handler method by attribute
  • Records idempotent business actions in business_action (per instance, per action code)
  • ACKs the engine via a transactional outbox (survives crashes)
  • Pins a handler version per workflow instance on first event
  • Sends heartbeats to keep the engine's liveness check passing
🚫 What you do not need to do
  • Call AckAsync yourself — just return AckOutcome
  • Poll the engine — the consumer loops do that
  • Handle retry backoff — the outbox loop retries failed ACKs
  • Register by definition ID — names are resolved from the engine at startup
Key difference from the raw engine API
When using the engine directly, you subscribe to engine.EventRaised and call engine.AckAsync(…) yourself. With the consumer library, you write a LifeCycleWrapper subclass, return AckOutcome from your handler, and the library handles everything else — polling, delivery tracking, outbox retry, and ACK.
Getting Started

Quick Start

  1. 1
    Decorate your class — apply [LifeCycleDefinition("my-workflow")] to a LifeCycleWrapper subclass.
  2. 2
    Add handler methods — decorate with [TransitionHandler(eventCode)] or [HookHandler("route")].
  3. 3
    Implement abstract methods — OnUnhandledTransitionAsync and OnUnhandledHookAsync for unregistered events.
  4. 4
    Register with DI — add services.AddTransient<MyHandler>().
  5. 5
    Build and start — build a WorkFlowConsumerManager, call RegisterAssembly, then StartAsync. If you are using DI, register WorkFlowConsumerService and let the hosted bootstrap initialize it.
// 1. Handler class
[LifeCycleDefinition("loan-approval")]
public class LoanApprovalHandler : LifeCycleWrapper {
    private readonly IEmailService _email;
    public LoanApprovalHandler(IEmailService email) { _email = email; }

    [TransitionHandler(100)]  // event code 100 = "submit"
    public async Task<AckOutcome> OnSubmitAsync(ILifeCycleTransitionEvent evt, ConsumerContext ctx) {
        await _email.SendConfirmationAsync(evt.EntityId, ctx.CancellationToken);
        return AckOutcome.Processed;
    }

    protected override Task<AckOutcome> OnUnhandledTransitionAsync(
        ILifeCycleTransitionEvent evt, ConsumerContext ctx)
        => Task.FromResult(AckOutcome.Processed);  // or Retry/Failed as needed

    protected override Task<AckOutcome> OnUnhandledHookAsync(
        ILifeCycleHookEvent evt, ConsumerContext ctx)
        => Task.FromResult(AckOutcome.Processed);
}

// 2. DI registration
services.AddTransient<LoanApprovalHandler>();

// 3. Build + start consumer
var consumer = await new WorkFlowConsumerMaker {
    EngineProxy     = inProcessFeed,
    ServiceProvider = serviceProvider,
    Options         = new WorkFlowConsumerOptions {
        ConsumerGuid = "my-app-consumer",
        EnvCode      = 1
    }
}.WithConnectionString(connectionString).Build(adapterGateway);
consumer.RegisterAssembly(Assembly.GetExecutingAssembly());
await consumer.StartAsync(cancellationToken);C#
Core Concepts

The Event Philosophy: Trigger vs Handler

The single most important thing to understand before writing consumer handlers is that the same event code does two completely different jobs depending on who uses it and when.

📤 TriggerAsync(event = 200)
  • Who calls it: your application code (or AutoTransitionAsync)
  • What it means: "please advance the state machine using event 200"
  • What the engine does: validates the transition, records it, unlocks the next state
  • When handlers run: after this returns — consumers are notified asynchronously
📥 [TransitionHandler(200)]
  • Who runs it: the consumer library, via the PollLoop
  • What it means: "event 200 was confirmed — run your business logic now"
  • What you do: execute side-effects (send emails, call APIs, update records)
  • ACK outcome: return AckOutcome.Processed when done
The key insight
TriggerAsync(event=200) and [TransitionHandler(200)] share the same integer code, but they are not two halves of a round-trip call. TriggerAsync is a command to the engine. The handler is the engine's notification back to you — and it may fire for this consumer even if a completely different service sent the trigger.

The Full Flow

Here is a concrete example: an external system submits a loan application (event 100), the consumer sends a confirmation email and then uses AutoTransitionAsync to advance to the next state (event 200 = "move to review").

Step 1 — External caller
  engine.TriggerAsync({ Event: "100", EntityGuid: "loan-xyz" })
        │
        ▼ Engine validates & records the state transition
        │  • Checks policy: is event 100 legal from current state?
        │  • Creates lifecycle entry, creates ack_consumer rows
        │  • Returns immediately — handler runs later
        │
Step 2 — Consumer PollLoop picks up the dispatch
  GetDueTransitionsAsync → [TransitionHandler(100)] fires
        │
        ▼ Handler runs business logic
        │  • Sends confirmation email (ExecuteBusinessActionAsync)
        │  • Calls AutoTransitionAsync(ctx, success: true)
        │    └─ records _nextEvent = ctx.OnSuccessEvent (200)
        │  • Returns AckOutcome.Processed
        │
Step 3 — Consumer writes outbox row, ACKs engine
  outbox.next_event = 200 (persisted)
  engine.AckAsync(ackGuid, Processed)
        │
        ▼ ACK confirmed — engine lifts its ACK gate for this instance
        │
Step 4 — Consumer fires the next trigger (post-ACK)
  engine.TriggerAsync({ Event: "200", InstanceGuid: "..." })
        │
        ▼ Engine records transition to "review" state
        │
Step 5 — PollLoop picks up event 200 dispatch
  [TransitionHandler(200)] fires → reviewer notification, etc.

Why the Same Event Code Appears Twice

Looking at event 100 above, it appears in both:

  • TriggerAsync({ Event: "100" }) — the request to the engine
  • [TransitionHandler(100)] — the engine's notification that the transition happened

This is intentional. The event code is the shared vocabulary between the engine's state machine and your business logic. The engine uses it to look up which transitions are legal; the consumer uses it to route to the right handler. The same concept, two operations separated in time.

ACK gate — applies to [TransitionHandler] only
The engine's ACK gate checks whether the most recent lifecycle (transition) ACK for the instance has been resolved before accepting a new trigger. The gate query joins lc_ack — it does not look at hook_ack.

[TransitionHandler]: calling Engine.TriggerAsync here is blocked — the current transition's lc_ack row is still pending. Use AutoTransitionAsync(ctx, success) instead — it defers the trigger until after ACK is confirmed.

[HookHandler]: calling Engine.TriggerAsync directly is safe — hook ACKs live in hook_ack which the gate never checks. The existing interactive hook wrappers exploit this intentionally.

AutoTransitionAsync — Built-in State Chaining

The policy attached to a workflow instance defines which event to fire on success and which to fire on failure (the complete.success and complete.failure fields). These are surfaced on ConsumerContext as OnSuccessEvent and OnFailureEvent. Instead of writing the trigger boilerplate yourself, call AutoTransitionAsync:

[TransitionHandler(100)]
public async Task<AckOutcome> OnSubmitAsync(ILifeCycleTransitionEvent evt, ConsumerContext ctx)
{
    var result = await ExecuteBusinessActionAsync(ctx, ACTION_SEND_CONFIRM,
        async (ct) => { await _email.SendConfirmationAsync(ctx.EntityGuid, ct); return null; });

    // Records intent to fire ctx.OnSuccessEvent (or OnFailureEvent) AFTER ACK.
    // Does NOT call the engine here — safe inside a handler.
    await AutoTransitionAsync(ctx, success: true);

    return AckOutcome.Processed;  // ACK outcome = delivery confirmed
}C#

Under the hood:

  1. AutoTransitionAsync sets _nextEvent on the wrapper instance to the resolved event code.
  2. After dispatch, the manager captures wrapper._nextEvent and persists it to outbox.next_event.
  3. After engine.AckAsync succeeds (inline or via the OutboxLoop), the manager calls engine.TriggerAsync({ Event: nextEvent.ToString() }).
  4. If that trigger fails, a NEXT_EVENT_FAILED notice is raised — the ACK is already confirmed and cannot be undone, but the engine's monitor will detect the stale state independently.
AckOutcome.Processed ≠ business success
AckOutcome.Processed means "I have received and processed this delivery — do not re-send it." It says nothing about whether the business logic succeeded or failed. AutoTransitionAsync(ctx, success: true/false) is what determines which next state the workflow moves to. The two are entirely orthogonal: you can AutoTransitionAsync(ctx, success: false) and still return AckOutcome.Processed, meaning "I handled this delivery; the workflow should now move to the failure branch."
Core Concepts

Architecture

Background Loops

WorkFlowConsumerManager.StartAsync launches three independent background loops on the thread pool. WorkFlowConsumerService is the hosted facade that builds and initializes the manager.

HeartbeatLoop
  • Calls engine.BeatConsumerAsync on a timer
  • Default interval: every 30 seconds
  • Keeps the consumer "alive" from the engine's perspective
  • If heartbeat goes stale, engine stops dispatching to this consumer
PollLoop
  • Calls engine.GetDueTransitionsAsync + GetDueHooksAsync
  • Dispatches each item via a semaphore-bounded concurrency pool
  • If no events: sleeps for PollInterval (default 5s)
  • If events found: polls again immediately
OutboxLoop
  • Retries ACKs that failed to reach the engine
  • Reads pending rows from outbox table where next_retry_at <= now
  • If empty: sleeps for OutboxInterval (default 15s)
  • Survives process crashes — outbox is durable
Startup sequence
  • Scan all loaded assemblies for [LifeCycleDefinition] classes
  • Query engine to resolve definition names → IDs
  • Register consumer with engine, receive assigned consumer ID
  • Start the three loops
Engine DB (ack_consumer)
    │
    ▼ PollLoop — GetDueTransitionsAsync / GetDueHooksAsync
WorkFlowConsumerManager
    │
    ├─ Semaphore (MaxConcurrency) ──► ProcessItemAsync
    │       ├─ Upsert instance row   (consumer DB: instance — keyed by instance_guid)
    │       ├─ Upsert inbox row      (consumer DB: inbox — keyed by ack_guid)
    │       ├─ Pin handler version   (first event for this instance only)
    │       ├─ LifeCycleWrapper.Dispatch ──► [TransitionHandler] / [HookHandler] method
    │       │       └─ Returns AckOutcome
    │       └─ Write outbox row ──► OutboxLoop ──► engine.AckAsync
    │
    └─ HeartbeatLoop ──► engine.BeatConsumerAsync

LifeCycleWrapper — The Handler Base Class

All workflow handlers inherit from LifeCycleWrapper. This base class provides business action helpers and the internal dispatch machinery. Subclasses are resolved from DI per event dispatch — constructor-inject your dependencies normally.

Business action helpers (protected)

MethodPurpose
ExecuteBusinessActionAsync(ctx, actionCode, action, mode?)Runs the action delegate and records the result in business_action. Skips re-execution if already completed (default mode). See Business Actions section.
IsBusinessActionCompletedAsync(ctx, actionCode)Returns true if a completed record already exists for (instanceId, actionCode). Does not execute the action.
GetBusinessActionAsync(ctx, actionCode)Returns the stored BusinessActionRecord (including result JSON) or null if none exists.
ReadDecisionFromResultJson(resultJson, defaultValue)Static helper — reads the "decision" bool field from a result JSON string.

State-chaining helper (protected)

MethodPurpose
AutoTransitionAsync(ctx, bool success)Records the next event to fire after ACK is confirmed. Does not call the engine — the manager fires the trigger post-ACK. Reads ctx.OnSuccessEvent or ctx.OnFailureEvent from the policy; throws InvalidOperationException if no matching event is defined. See Event Philosophy.

Engine access (protected)

Use the Engine property inside handler methods to call the engine (trigger events, fetch timelines, read/write instance context, etc.). It is injected by the consumer manager before dispatch — do not inject ILifeCycleExecution via the constructor.
[HookHandler] methods may call Engine.TriggerAsync directly — hook ACKs are not checked by the engine's ACK gate.
[TransitionHandler] methods must not call Engine.TriggerAsync directly — the current transition's ACK is still pending and the gate will reject it. Use AutoTransitionAsync instead. See the Event Philosophy section.

// Inside a handler method:
await Engine.TriggerAsync(new LifeCycleTriggerRequest { /* ... */ }, ctx.CancellationToken);C#

Required abstract methods

Implement these to handle events that have no matching [TransitionHandler] or [HookHandler] attribute:

protected abstract Task<AckOutcome> OnUnhandledTransitionAsync(
    ILifeCycleTransitionEvent evt, ConsumerContext ctx);

protected abstract Task<AckOutcome> OnUnhandledHookAsync(
    ILifeCycleHookEvent evt, ConsumerContext ctx);C#
Do not return Processed by default blindly
If you return Processed for all unhandled events, silently-ignored events will never re-deliver. Consider logging an alert and returning Failed for truly unexpected events, or Processed only if your handler deliberately handles all events.

Instance & Entity Model

Every workflow instance flowing through the consumer has two identifiers that appear in ConsumerContext and in the local instance table:

IdentifierSourceCharacteristics
instance_guid (ctx.InstanceGuid) Engine-assigned Stable UUID for one specific workflow run. Unique across the engine DB. Never changes. Used as the consumer's deduplication key when creating the local instance row.
entity_guid (ctx.EntityGuid) Application-supplied The application's business key — e.g. "order-12345", "user-abc". Passed to the engine at trigger time as external_ref. Forwarded on every event. Multiple workflow instances can share the same entity_guid.

Why the consumer needs an instance table

The engine identifies instances by UUID string. String primary keys are inefficient for FK joins in an operational DB with high event throughput. The consumer's instance table creates a local bigint surrogate ID for each engine instance, enabling fast indexed joins across all consumer tables (inbox, outbox, business_action, inbox_action).

The row is created lazily on the first event for a given engine instance — the consumer does not know about the instance until it receives an event. An INSERT IGNORE pattern ensures the row is created at most once, even under concurrent delivery.

ColumnDescription
idLocal surrogate bigint PK — FK target for all other consumer tables
guidThe engine's instance_guid — unique; used as upsert deduplication key
entity_guidThe application's business key — stored for query filtering (find all inbox rows for an order ID)
def_nameDefinition name (e.g. "loan-approval")
def_version_valueActual engine definition version value mirrored into the consumer instance row
createdEngine-side instance creation timestamp mirrored into the consumer database

How entity_guid flows through the system

  1. 1
    Application triggers a workflow — calls engine.TriggerAsync(new LifeCycleTriggerRequest { EntityGuid = "order-12345", DefName = "loan-approval", ... }). The engine records this as instance.external_ref.
  2. 2
    Engine fires events — every dispatched transition or hook event carries both InstanceGuid (the engine's stable UUID) and EntityGuid (the application key) in its payload.
  3. 3
    Consumer upserts the instance row — on first event, inserts instance(guid, entity_guid, def_name, ...). Returns the local instanceId bigint, which becomes the FK for the inbox row and any business_action rows.
  4. 4
    Handler accesses both keys — use ctx.EntityGuid to look up application records, ctx.InstanceGuid for engine calls (e.g. Engine.GetTimelineAsync), and ctx.InstanceId if you need the local surrogate for direct consumer DB queries.
Business actions are scoped to one instance, not to entity_guid
ExecuteBusinessActionAsync is keyed by (instance_id, action_code). If the same entity has two separate workflow instances (e.g. two loan applications for the same customer), their business actions are completely independent. Each instance carries its own idempotency envelope.
entity_guid is opaque — the consumer does not validate it
The consumer stores entity_guid exactly as received from the engine event. It is the application's responsibility to choose stable, unique-enough keys and to handle multiple instances per entity when querying (e.g. a customer may have multiple active loan applications each with the same entity_guid but different instance_guids).

RunCount & Hook Reruns NEW

Every hook dispatch now carries a RunCount — how many times the engine has fully dispatched this hook to consumers, including the current delivery. This lets handlers detect reruns caused by ReopenAsync or workflow loops through the same state.

Where RunCount comes from
  • Engine computes: COUNT(*) FROM hook_lc WHERE hook_id = h.id AND dispatched = 1
  • Included in the engine's dispatch payload (ILifeCycleHookEvent.RunCount)
  • Consumer stores it in the inbox.run_count column at upsert time
  • Exposed on ConsumerContext.RunCount — no cast required
When RunCount > 1
  • A previous run of this hook reached consumers (dispatched = 1 in hook_lc)
  • Workflow may have looped back to the same state via ReopenAsync
  • Side effects from the first run may already have happened
  • Use ExecuteBusinessActionAsync to skip already-completed side-effects

Accessing RunCount

private const int ACTION_SEND_EMAIL = 1;

[HookHandler("NOTIFY.EMAIL")]
public async Task<AckOutcome> OnNotifyAsync(ILifeCycleHookEvent evt, ConsumerContext ctx) {

    // Option A — from ConsumerContext (no cast needed, always available)
    if (ctx.RunCount > 1) {
        _log.LogWarning("Hook re-run #{N} — prior email may have been sent", ctx.RunCount);
    }

    // Option B — from the event itself (same value)
    if (evt.RunCount > 1) { /* ... */ }

    // ExecuteBusinessActionAsync prevents re-sending on re-delivery (same ack_guid)
    // and also preserves the result across process restarts.
    await ExecuteBusinessActionAsync(ctx, ACTION_SEND_EMAIL, async (ct) => {
        await _mailer.SendAsync(ctx.EntityGuid, ct);
        return null;
    });
    return AckOutcome.Processed;
}C#
RunCount vs re-delivery
RunCount > 1 means the engine triggered the hook a second time (new lifecycle entry, new ack_guid). This is different from re-delivery of the same dispatch (same ack_guid, same RunCount). Re-delivery happens when the consumer crashes or returns Retry. Use ExecuteBusinessActionAsync to guard against both re-delivery and genuine reruns — it is keyed by (instance_id, action_code) which is stable across both scenarios.

hook_lc — the engine design behind RunCount

The engine separates hook definitions (one row per emission slot) from per-run dispatch records:

TableOne row perRole
hook(instance, state, route, on_entry)Static definition. Never deleted on reopen.
hook_lc(hook, lifecycle entry)Per-run dispatch record. dispatched flips to 1 after ACK rows are created. RunCount = COUNT(*) WHERE dispatched=1.

When ReopenAsync is called, hook rows are kept. Each new lifecycle entry creates fresh hook_lc rows with dispatched=0. Old hook_lc rows stay as history, which is how RunCount accumulates across reruns.

WrapperRegistry — Auto-Discovery

The registry maps definition IDs (engine-assigned) to handler types. It works in two phases:

  1. 1
    Assembly scan (RegisterAssembly) — finds all non-abstract types inheriting LifeCycleWrapper with [LifeCycleDefinition]. Stores them by definition name.
  2. 2
    Engine resolution (on WorkFlowConsumerManager.StartAsync) — for each discovered name, calls engine.GetDefinitionIdAsync. Binds the returned def_id to the handler type.

At dispatch time, TryGetRegistration(defId) looks up the handler type. If no registration exists for an incoming definition ID for this consumer, the consumer rejects that delivery terminally with AckOutcome.Failed and sends the reason back to the engine.

Multiple assemblies
Call RegisterAssembly for each assembly that contains handler classes before StartAsync. WorkFlowConsumerManager.StartAsync also auto-scans all currently loaded assemblies as a fallback. 
consumer
    .RegisterAssembly(Assembly.GetExecutingAssembly())
    .RegisterAssembly("OtherHandlers.Assembly");  // name or already-loaded assembly
await consumer.StartAsync(ct);C#

Dispatch Flow (ProcessItemAsync)

When an event item is dequeued from the engine, ProcessItemAsync runs the following steps:

  1. 1
    Upsert instance row — finds or creates a row in the instance table keyed by the engine's instance_guid. Returns a local bigint instanceId surrogate key used as FK for all subsequent rows.
  2. 2
    Upsert inbox row — inserts a row in the inbox table keyed by ack_guid (unique). Stores kind, route, event code, run_count, etc. Returns inboxId.
  3. 3
    Pin handler version (first event for this instance only) — queries inbox for the earliest row with a handler_version for the same instance_id. Pins it permanently on the current inbox row.
  4. 4
    Build context — creates ConsumerContext with InboxId, InstanceId, InstanceGuid, EntityGuid, AckGuid, HandlerVersion, RunCount, CancellationToken, etc.
  5. 5
    Resolve handler from DI — resolves the wrapper type from IServiceProvider. Startup validation now fails fast if a wrapper needs constructor injection but is not registered. A parameterless-constructor fallback is still allowed for simple wrappers. Injects business action DAL and inbox action DAL.
  6. 6
    Dispatch — calls wrapper.DispatchTransitionAsync or wrapper.DispatchHookAsync. The dispatcher selects the best-matching handler method by MinVersion. Returns AckOutcome.
  7. 7
    Write outbox + attempt ACK — captures wrapper._nextEvent (set by AutoTransitionAsync if called). Writes outcome + next_event to outbox table. Attempts to call engine.AckAsync immediately. On success: fires the next trigger if next_event is set. If ACK fails: sets next_retry_at and the OutboxLoop retries later — it too fires the next trigger after ACK is confirmed.
Acknowledgement is mandatory
Every consumer delivery now carries an ack_guid. The consumer always records the inbox row, writes the outbox row, and confirms the ACK back to the engine. Deliveries with a missing ack_guid are treated as invalid and skipped with a notice.

Inbox, Outbox, and Retry Model

The consumer runtime is proxy-driven, not engine-implementation-driven. The consumer manager only talks to ILifeCycleEngineProxy. That proxy may drain in-process channels, call HTTP endpoints, or use any other transport. The consumer manager does not care how items arrive.

Current in-process behaviour
In the current in-process setup, the proxy does not query the engine DB for due items. The engine raises events through EventRaised, the proxy buffers them in memory channels, and the consumer poll loop drains those channels via GetDueTransitionsAsync / GetDueHooksAsync.

What each consumer table means

TableMeaningCreated / updated when
instanceLocal mirror of the engine instance so all consumer joins use a local bigint key.Created or reused before the inbox row is written.
inboxCanonical record that this delivery was received by the consumer.Created as soon as the consumer accepts a valid dispatched item. Keyed by ack_guid.
inbox_statusProcessing state of that delivery.Moves through Received → Processing → Processed or Failed while the wrapper runs.
outboxQueue entry for the ACK outcome that must be delivered back to the engine.Created only after the wrapper finishes and an AckOutcome is known.
outbox_historyAttempt ledger for ACK sends.Written each time the consumer confirms or fails an ACK send attempt.

When the outbox is created

The outbox row is not created when the event first arrives. First the consumer records the delivery in inbox, runs the wrapper, and decides an AckOutcome. Only then does it write the outbox row. This is why the correct mental model is:

  • inbox = "I received this delivery"
  • outbox = "I have decided what ACK outcome to send back"

Failure paths

Handler or processing failure
  • The wrapper throws or business logic cannot complete
  • inbox_status becomes Failed
  • The consumer still writes an outbox row with AckOutcome.Retry
  • If that ACK reaches the engine, the engine re-schedules the same delivery later
ACK transport failure
  • The wrapper already finished and the ACK outcome is already decided
  • The consumer writes outbox, then engine.AckAsync fails
  • outbox.status stays Pending and next_retry_at is scheduled
  • The wrapper is not run again by this retry path; only the ACK send is retried
Only one case is a true skip
If the incoming item's consumerId does not match this running consumer instance, the consumer does nothing. That delivery is not addressed to this consumer, so it must not ACK, fail, or mutate engine state for it. All other same-consumer contract problems should be surfaced explicitly rather than ignored.
If local DB persistence fails before ACK confirmation
If the consumer cannot finish its local inbox or outbox persistence work, there is no confirmed ACK back to the engine. In that case the consumer has nothing durable to continue from locally, so the engine monitor eventually re-delivers the item through its normal ACK scheduling rules.

   └──► consumer ACKs engine with Retry
          └──► engine schedules re-delivery later

if ACK send fails after wrapper completed
   └──► consumer keeps outbox Pending
          └──► OutboxLoop retries the ACK send later

Business Actions — Persistent Idempotent Operations

Business actions record that a significant operation was performed for a workflow instance and guarantee it is never re-executed — even across process restarts, re-deliveries, or multiple event types. The key is (instance_id, action_code) — one surrogate instance row, one integer action code.

Why business actions instead of inline re-delivery guards?
At-least-once delivery means your handler may run more than once for the same logical event (re-delivery, process restart). Business actions let you wrap any external side-effect — email send, payment charge, API call — so that on re-delivery the delegate is simply skipped and the cached result returned. Unlike a simple boolean flag, the result JSON is preserved across restarts.

Primary API

Usage Example

private const int ACTION_CREDIT_CHECK = 10;  // integer action code — define as const

[TransitionHandler(200)]  // approve event

{
    // Runs ONCE per workflow instance, regardless of re-deliveries or process restarts.
    var result = await ExecuteBusinessActionAsync(
        ctx,
        ACTION_CREDIT_CHECK,

            var approved = await _creditService.CheckAsync(ctx.EntityGuid, ct);
            // Return an object — serialised to JSON automatically.
            // ReadDecisionFromResultJson reads the "decision" field later.
            return new { decision = approved };

    // result.AlreadyCompleted == true on re-delivery — delegate was NOT called again.
    bool approved = ReadDecisionFromResultJson(result.ResultJson, defaultValue: false);

    return AckOutcome.Processed;

Action codes are integers — scope them to your handler class
Action codes are arbitrary positive integers meaningful only within a handler class. Define them as private constants (e.g. const int ACTION_SEND_EMAIL = 1). The consumer DB stores them as business_action.action_code alongside the instance_id FK.

Handler Versioning

Each workflow instance has a handler version pinned on its first event. This lets you deploy new handler logic without breaking in-flight instances running on older code.

How pinning works

  1. First event for a workflow instance arrives. The service queries inbox for any earlier row that already has a handler_version for the same instance_id. If found, that version is reused for consistency.
  2. The handler version is stored on the inbox row and never changes for that instance (unless HandlerUpgrade = AllowUpgrade).
  3. On every subsequent event for the same instance, the pinned version is loaded from inbox into ctx.HandlerVersion.

MinVersion picker algorithm

For an event, the dispatcher collects all methods decorated with a matching attribute and picks the one with the highest MinVersion that is ≤ ctx.HandlerVersion.

// Example: three versions of the same event handler
[TransitionHandler(100, minVersion: 0)]   // selected when HandlerVersion = 0 or 1
public Task<AckOutcome> OnSubmitV1Async(...) { ... }

[TransitionHandler(100, minVersion: 2)]   // selected when HandlerVersion = 2 or 3
public Task<AckOutcome> OnSubmitV2Async(...) { ... }

[TransitionHandler(100, minVersion: 4)]   // selected when HandlerVersion >= 4
public Task<AckOutcome> OnSubmitV4Async(...) { ... }C#

default_handler_upgrade policy

The DefaultHandlerUpgrade option in WorkFlowConsumerOptions controls whether existing in-flight instances pick up newer handler methods when you deploy updated code.

ValueBehaviourUse case
Pinned (default)Version is set once at first event delivery for the workflow instance and never changes. New code deployments do not affect in-flight instances — they continue using the handler methods that were selected when they started.Safest option. Use when you want full isolation between old and new instances — ship improved logic for new instances while old in-flight instances keep their original handler path.
AutoThe effective version advances with the wrapper's declared maximum version. Existing in-flight instances automatically pick up new handler methods on next event delivery after a deployment.Use when all handler versions are backward-compatible and you want every instance to benefit from fixes immediately, regardless of when they started.
 
// Set upgrade policy in options:

    DefaultHandlerUpgrade = HandlerUpgrade.Pinned  // default — safe for production

// Example: instance pinned at version 2, two handlers defined
[TransitionHandler(100, minVersion: 1)]  // selected: HandlerVersion=2 ≥ minVersion=1, highest match

Scenario: ship improved logic without breaking in-flight instances
You have 500 loan applications in the UnderReview state. You deploy a new handler at minVersion: 3 with improved credit-check logic. With Pinned, all 500 existing workflow instances continue using their pinned version (e.g. version 2, selecting the minVersion: 1 handler). New workflow instances created after the deployment will get version 3 and use the new handler path. Neither group is disrupted.

Outbox & Reliability

The consumer uses a transactional outbox to guarantee ACK delivery even if the process crashes between completing work and sending the ACK to the engine.

Happy path
  • Handler runs, returns AckOutcome
  • Consumer writes outbox row (Pending)
  • Consumer calls engine.AckAsync immediately
  • On success: outbox row → Confirmed
ACK failure path
  • Handler completes but engine.AckAsync fails (network, timeout)
  • Outbox row stays Pending with next_retry_at
  • OutboxLoop retries on schedule (OutboxRetryDelay)
  • Each attempt is recorded in outbox_history
Crash safety
If the process crashes after the handler completes but before ACK is sent, the outbox row is durable in the consumer DB. When the process restarts, the OutboxLoop will pick up the pending row and ACK the engine. The engine will not re-deliver the event to this consumer as long as the ACK arrives within the engine's retry window.
Architecture

Engine Proxy

The consumer package has no dependency on the engine package. No consumer-side code ever holds a reference to IWorkFlowEngine or IWorkFlowEngineAccessor. All engine interaction flows through two layered interfaces — one for wrapper handlers, one for consumer infrastructure.

Two-level access boundary
Wrapper handlers receive ILifeCycleExecution — everything they need to call the engine (trigger, timeline, runtime log, etc.) with no access to infrastructure-level polling.

Consumer infrastructure (manager, service, DI) receives ILifeCycleEngineProxy — extends ILifeCycleExecution with the polling methods used exclusively by the internal dispatch loop.

Interface hierarchy

InterfaceExtendsWho receives itExtra surface
ILifeCycleExecution ILifeCycleConsumerBus + ILifeCycleRuntimeBus Wrapper handlers via LifeCycleWrapper.Engine None — this is the complete handler surface
ILifeCycleEngineProxy ILifeCycleExecution Consumer manager, consumer service, DI GetDueTransitionsAsync, GetDueHooksAsync, NoticeRaised

How wrappers get engine access

Wrappers do not receive ILifeCycleExecution via constructor injection. The consumer manager sets it as an internal field immediately before dispatch — exactly like _stepDal and _businessActionDal. Wrapper constructors only carry the wrapper's own application services (repositories, HTTP clients, etc.). 

// Wrapper constructor — only app services, nothing from the framework:
public MyWrapper(IMyRepository repo) { /* ... */ }

// Inside a handler method — Engine is injected before dispatch by the manager:
[HookHandler("APP.MY.HOOK")]
private async Task<AckOutcome> OnMyHookAsync(ILifeCycleHookEvent evt, ConsumerContext ctx) {
    await Engine.TriggerAsync(new LifeCycleTriggerRequest { /* ... */ }, ctx.CancellationToken);
    return AckOutcome.Processed;
}C#

Why wrappers cannot call GetDueTransitionsAsync

GetDueTransitionsAsync and GetDueHooksAsync drain the in-memory delivery channels that the engine's EventRaised event writes into. If wrapper code could call those methods, it would silently steal events from the dispatch pipeline — events the consumer manager would never see. Because wrappers only receive ILifeCycleExecution (which does not have those methods), this is impossible by design.

Proxy implementations

ImplementationPackageWhen to use
InProcessEngineProxy Engine package Engine and consumer run in the same .NET host. Direct method calls to the engine; events buffered in in-memory channels. Engine resolved lazily on first call.
HttpEngineProxy (future) Consumer package Engine runs on a separate host. Every method issues an HTTP call to the engine's REST API. Consumer and wrapper code are unchanged.

Event delivery vs on-demand queries

Not all methods on the proxy work the same way:

📥 Event delivery (push → poll)

GetDueTransitionsAsync / GetDueHooksAsync

Exist because the engine has EventRaised. The proxy buffers those events (in-process: in-memory channels; HTTP: server-side buffer). The consumer manager's poll loop drains them every tick.

Consumer infrastructure only. Wrapper handlers cannot call these.

🔍 On-demand (consumer initiates)

All ILifeCycleRuntimeBus methods

The consumer (or a wrapper handler) calls these at will — no engine event triggered them. Examples: trigger next state, fetch timeline JSON, upsert runtime status, suspend or reopen an instance.

In-process: direct engine call. HTTP: POST/GET to engine REST endpoint.

Do not confuse
ListPendingAcksAsync is an on-demand query — the consumer asking "what's unresolved right now?". GetDueTransitionsAsync is event delivery — draining events the engine already pushed into the proxy's buffer. They are fundamentally different operations.

DI registration (in-process host)

Register the proxy between the engine and consumer service calls. The engine package provides the AddInProcessEngineProxy() extension: 

// 1. Engine — registers IWorkFlowEngineAccessor:
services.AddWorkFlowEngineService(configuration);

// 2. Proxy — engine package; lazily resolves engine on first call:
services.AddInProcessEngineProxy();

// 3. Consumer — sees only ILifeCycleEngineProxy, never engine types:
services.AddWorkFlowConsumerService(configuration);C#

For a remote deployment, skip step 2 and register an HttpEngineProxy instead. Steps 1 and 3 are identical in both topologies.

Consumer Notice Codes

Subscribe to manager.NoticeRaised to receive operational events from the consumer framework. These are distinct from the engine's notice codes — they are fired by the consumer library itself during polling, dispatch, and outbox operations.

CodeSeverityMeaning
HEARTBEAT_ERROR Error Heartbeat call to the engine (BeatConsumerAsync) failed. If this persists beyond the engine's ConsumerTtlSeconds, the engine will stop dispatching events to this consumer.
POLL_ERROR Error Unhandled exception in the poll loop. The loop will sleep and retry. Individual dispatch errors are reported separately via DISPATCH_ERROR.
OUTBOX_ERROR Error Unhandled exception in the outbox loop. Pending ACKs will not be retried until the loop recovers on its next cycle.
OUTBOX_ACK_FAILED Error A specific outbox entry failed to reach the engine after the maximum number of retry attempts. The engine may not receive the ACK for this event — manual investigation may be required.
DISPATCH_SCHEDULE_ERROR Error Failed to schedule an event item for dispatch (e.g. semaphore acquisition error). The engine's monitor will re-send the item after its next_due interval elapses.
DISPATCH_ERROR Error Unhandled exception in ProcessItemAsync — thrown outside the handler delegate itself. The event will be re-delivered on the next poll cycle.
DISPATCH_INVALID_ACK Error An event arrived for this consumer with a missing or null ack_guid. This is a contract violation: the consumer cannot confirm, fail, or cancel anything on the engine without that key. The event is rejected locally with a notice only.
WRAPPER_ERROR Error The LifeCycleWrapper dispatch method threw an unhandled exception. The outcome is automatically set to Retry so the engine re-delivers the event.
REGISTRY_MISS Warn The consumer received an event for this consumer, but there is no registered wrapper for the incoming definition ID. Three Background Loops — Detailed Reference

WorkFlowConsumerManager.StartAsync launches three independent background loops. Each runs on a separate thread-pool task and does not share state with the others.

♥ HeartbeatLoop
  • Calls engine.BeatConsumerAsync(consumerGuid, envCode, ttlSeconds) every heartbeat_interval
  • Default interval: HeartbeatInterval option (default 30 seconds)
  • Fires HEARTBEAT_ERROR notice on failure — does not crash the loop
  • If this loop stalls or errors for longer than the engine's ConsumerTtlSeconds, the engine marks the consumer as down and stops dispatching events to it
  • Keep HeartbeatInterval comfortably below ConsumerTtlSeconds (e.g. 30s interval with 120s TTL)
⟳ PollLoop
  • Calls engine.GetDueTransitionsAsync and engine.GetDueHooksAsync in sequence
  • This is the pull path — it catches events that the engine's push (EventRaised) may have missed (e.g. engine in-process delivery skipped)
  • Dispatches each item via a SemaphoreSlim-bounded concurrency pool (MaxConcurrency option)
  • If events found: polls again immediately without sleeping
  • If no events: sleeps for PollInterval (default 5 seconds) before next poll
  • Fires POLL_ERROR notice on unhandled exception — sleeps and retries
📤 OutboxLoop
  • Scans outbox rows with status Pending where next_retry_at <= now
  • For each pending row: calls engine.AckAsync with the stored outcome
  • On success: marks row Confirmed. On failure: updates next_retry_at and increments retry count
  • After max retries: fires OUTBOX_ACK_FAILED notice
  • If no pending rows: sleeps for OutboxInterval (default 15 seconds)
  • Guarantees at-least-once ACK delivery even when the engine is temporarily unreachable
  • Survives process crashes — outbox rows are durable in the consumer DB
▶ Startup Sequence
  • Scan registered assemblies for [LifeCycleDefinition] classes
  • Query engine to resolve definition names to IDs — logs REGISTRY_RESOLVE_FAILED for any unresolved names
  • Register consumer with engine via RegisterConsumerAsync
  • Launch HeartbeatLoop, PollLoop, and OutboxLoop concurrently
HeartbeatLoop ──────────────────────────────────────► BeatConsumerAsync (every heartbeat_interval)

PollLoop ─── GetDueTransitionsAsync ─┐
         └── GetDueHooksAsync ────────┼─► Semaphore ──► ProcessItemAsync ──► OutboxRow
                                      │                        │
                                      │              ┌─────────▼──────────┐
                                      │              │ handler.DispatchAsync│
                                      │              │ returns AckOutcome  │
                                      │              └─────────────────────┘
                                      └─ (sleep PollInterval if empty)

OutboxLoop ─── scan outbox (Pending, next_retry_at <= now)
           └──► engine.AckAsync ──► Confirmed / next_retry_at bumped

Example 2 — Transition Handler with AutoTransitionAsync

The policy defines which event to fire on success (complete.success) and failure (complete.failure). These are surfaced as ctx.OnSuccessEvent / ctx.OnFailureEvent. Use AutoTransitionAsync to record the next event intent — the consumer manager fires the trigger after ACK is confirmed, safely past the engine's ACK gate.

Do not call Engine.TriggerAsync inside a [TransitionHandler]
The engine's ACK gate checks pending lc_ack rows for the instance. When your handler is running, the current transition's ACK row is still pending — calling TriggerAsync here returns Applied=false, Reason="BlockedByPendingAck". AutoTransitionAsync stores the intent; the manager fires the trigger after ACK succeeds. This is safe in [HookHandler] methods — hook ACKs are not checked by the gate.
LifeCycleTriggerRequest.Metadata
When creating a new workflow instance, set Metadata to stamp an immutable tag on the instance. It is forwarded on every subsequent event via evt.Metadata. Example: Metadata = "env=prod,tier=gold". On follow-up trigger calls the field is ignored — metadata is set once at creation only. 
[LifeCycleDefinition("loan-approval")]
public class LoanApprovalHandler : LifeCycleWrapper {
    private readonly IReviewService _review;

    // Engine is provided by the base class — do NOT inject ILifeCycleExecution here
    public LoanApprovalHandler(IReviewService review) { _review = review; }

    [TransitionHandler(100)]  // submit → UnderReview
    public async Task<AckOutcome> OnSubmitAsync(
        ILifeCycleTransitionEvent evt, ConsumerContext ctx) {

        // Read policy params (from policy JSON "params" block)
        var slaHours = evt.Params?["sla_hours"]?.ToString() ?? "48";

        // Perform business logic
        var approved = await _review.RunAsync(evt.EntityId, slaHours, ctx.CancellationToken);

        // Record intent to trigger the next state AFTER this ACK is confirmed.
        // Reads ctx.OnSuccessEvent or ctx.OnFailureEvent (int?) from the policy.
        // Throws InvalidOperationException if the policy has no matching event defined.
        // DO NOT call Engine.TriggerAsync here — ACK gate will block it.
        await AutoTransitionAsync(ctx, success: approved);

        // AckOutcome.Processed = "delivery confirmed".
        // This is independent of whether the business outcome was success or failure.
        return AckOutcome.Processed;
    }

    protected override Task<AckOutcome> OnUnhandledTransitionAsync(
        ILifeCycleTransitionEvent evt, ConsumerContext ctx)
        => Task.FromResult(AckOutcome.Processed);
    protected override Task<AckOutcome> OnUnhandledHookAsync(
        ILifeCycleHookEvent evt, ConsumerContext ctx)
        => Task.FromResult(AckOutcome.Processed);
}C#

Example 3 — Business Actions (Idempotent Operations)

Use ExecuteBusinessActionAsync to wrap external side-effects. On re-delivery the delegate is skipped automatically and the cached result returned. Action codes are private integer constants — scoped to the handler class.

// ── Action code constants ──────────────────────────────────
private const int ACTION_VALIDATE = 1;
private const int ACTION_RESERVE  = 2;
private const int ACTION_NOTIFY   = 3;

[TransitionHandler(100)]
public async Task<AckOutcome> OnSubmitAsync(
    ILifeCycleTransitionEvent evt, ConsumerContext ctx) {

    // ── Action 1: Validate ────────────────────────────────────
    // On re-delivery this is skipped if already completed.
    // Exceptions inside the delegate propagate out — return Retry / Failed accordingly.
    var validateResult = await ExecuteBusinessActionAsync(ctx, ACTION_VALIDATE,
        async (ct) => {
            await _validator.CheckAsync(ctx.EntityGuid, ct);
            return null;  // no result payload needed
        });

    // ── Action 2: Reserve inventory ───────────────────────────
    var reserveResult = await ExecuteBusinessActionAsync(ctx, ACTION_RESERVE,
        async (ct) => {
            // Returns reservation ID — stored in result_json for later reference
            return await _inventory.ReserveAsync(ctx.EntityGuid, ct);
        });

    // ── Action 3: Notify applicant ────────────────────────────
    await ExecuteBusinessActionAsync(ctx, ACTION_NOTIFY,
        async (ct) => {
            await _email.SendSubmittedAsync(ctx.EntityGuid, ct);
            return null;
        });

    return AckOutcome.Processed;
}C#
On re-delivery
If the process crashes after Action 2 completes but before Action 3, the engine re-delivers the event. Actions 1 and 2 are skipped (already Completed in business_action for this instance_id). Only Action 3 executes. The per-delivery audit trail is recorded in inbox_action.
Exceptions inside the delegate propagate out
If the delegate throws, the exception is caught, the business_action row is marked Failed, and the exception is re-thrown. Catch it outside ExecuteBusinessActionAsync and return AckOutcome.Retry or Failed as appropriate. On the next delivery the action will be re-attempted (status is not Completed).

Example 4 — Hook Handler

Handle hook events emitted by the engine's policy. The route name in [HookHandler] must match the "event" field in the policy emit entry exactly.

[LifeCycleDefinition("loan-approval")]
public class LoanHookHandler : LifeCycleWrapper {
    private readonly IEmailService  _email;
    private readonly ICreditService _credit;

    public LoanHookHandler(IEmailService email, ICreditService credit) {
        _email = email; _credit = credit;
    }

    // Handles: "emit": [{ "event": "app.loan.notify_reviewer", ... }]
    [HookHandler("app.loan.notify_reviewer")]
    public async Task<AckOutcome> OnNotifyReviewerAsync(
        ILifeCycleHookEvent evt, ConsumerContext ctx) {
        // Params come from the policy "params" block
        var email = evt.Params?["reviewer_email"]?.ToString();
        if (string.IsNullOrWhiteSpace(email)) return AckOutcome.Failed;

        await _email.SendReviewRequestAsync(email, evt.EntityId, ctx.CancellationToken);
        return AckOutcome.Processed;
    }

    // Handles: "emit": [{ "event": "app.loan.run_credit_check", "blocking": true }]
    [HookHandler("app.loan.run_credit_check")]
    public async Task<AckOutcome> OnCreditCheckAsync(
        ILifeCycleHookEvent evt, ConsumerContext ctx) {
        try {
            var passed = await _credit.CheckAsync(evt.EntityId, ctx.CancellationToken);
            // Return Retry if transient; Processed/Failed for permanent outcomes.
            // If Failed on a blocking hook, engine may suspend the instance.
            return passed ? AckOutcome.Processed : AckOutcome.Failed;
        } catch {
            return AckOutcome.Retry;
        }
    }

    // Non-blocking audit log hook — failure is safe
    [HookHandler("app.loan.audit_log")]
    public Task<AckOutcome> OnAuditLogAsync(
        ILifeCycleHookEvent evt, ConsumerContext ctx) {
        _logger.LogInformation("Loan {id} entered review state", evt.EntityId);
        return Task.FromResult(AckOutcome.Processed);
    }

    protected override Task<AckOutcome> OnUnhandledTransitionAsync(
        ILifeCycleTransitionEvent evt, ConsumerContext ctx)
        => Task.FromResult(AckOutcome.Processed);
    protected override Task<AckOutcome> OnUnhandledHookAsync(
        ILifeCycleHookEvent evt, ConsumerContext ctx)
        => Task.FromResult(AckOutcome.Processed);
}C#

Example 5 — Handler Versioning

Deploy new logic for the same event code while keeping in-flight entities on their pinned version. Old entities continue using OnSubmitV1Async; new entities (with higher pinned versions) use the newer methods.

[LifeCycleDefinition("loan-approval")]
public class LoanApprovalHandler : LifeCycleWrapper {

    // ── Dispatch rules for event code 100 ──────────────────────────
    // HandlerVersion 0-1: use V1 (legacy flow)
    // HandlerVersion 2-3: use V2 (enhanced validation added)
    // HandlerVersion 4+ : use V4 (new credit pre-check)
    // The dispatcher picks the highest MinVersion that is ≤ HandlerVersion.

    [TransitionHandler(100, minVersion: 0)]
    public async Task<AckOutcome> OnSubmitV1Async(
        ILifeCycleTransitionEvent evt, ConsumerContext ctx) {
        // Original logic — existing in-flight entities still run this
        await _validator.BasicCheckAsync(evt.EntityId);
        return AckOutcome.Processed;
    }

    [TransitionHandler(100, minVersion: 2)]
    public async Task<AckOutcome> OnSubmitV2Async(
        ILifeCycleTransitionEvent evt, ConsumerContext ctx) {
        // New entities (HandlerVersion >= 2) run enhanced validation
        await _validator.EnhancedCheckAsync(evt.EntityId);
        await _notifier.SendReceiptAsync(evt.EntityId);
        return AckOutcome.Processed;
    }

    [TransitionHandler(100, minVersion: 4)]
    public async Task<AckOutcome> OnSubmitV4Async(
        ILifeCycleTransitionEvent evt, ConsumerContext ctx) {
        // Entities created after version 4 also run the credit pre-check
        await _validator.EnhancedCheckAsync(evt.EntityId);
        await _credit.PreCheckAsync(evt.EntityId);
        await _notifier.SendReceiptAsync(evt.EntityId);
        return AckOutcome.Processed;
    }

    // ── AllowUpgrade: let existing entities pick up new logic ──────
    // Set in WorkFlowConsumerOptions.DefaultHandlerUpgrade = AllowUpgrade
    // Entity with HandlerVersion=1 + AllowUpgrade will use V4 (max declared).
    // Only do this when the new logic is backwards-compatible.

    protected override Task<AckOutcome> OnUnhandledTransitionAsync(
        ILifeCycleTransitionEvent evt, ConsumerContext ctx)
        => Task.FromResult(AckOutcome.Processed);
    protected override Task<AckOutcome> OnUnhandledHookAsync(
        ILifeCycleHookEvent evt, ConsumerContext ctx)
        => Task.FromResult(AckOutcome.Processed);
}C#
Version pinning in practice
On first event for a new workflow instance, the consumer queries inbox for any existing pinned version for the same instance_id. This means all events for one instance are processed consistently — even when new handler versions are deployed mid-flight.

Example 6 — Hook Reruns & RunCount NEW

A workflow has a hook that sends a reviewer notification email. When the application calls ReopenAsync to restart a stalled loan, the hook fires again. The handler uses ctx.RunCount to log the rerun and ExecuteBusinessActionAsync to ensure the email is sent at most once per workflow instance per action code — regardless of re-delivery or reopen.

[LifeCycleDefinition("loan-approval")]
public class LoanHookHandler : LifeCycleWrapper {
    private readonly IEmailService _email;
    private readonly ILogger       _log;
    private const int STEP_SEND_EMAIL = 1;

    public LoanHookHandler(IEmailService email, ILogger<LoanHookHandler> log) {
        _email = email; _log = log;
    }

    [HookHandler("app.loan.notify_reviewer")]
    public async Task<AckOutcome> OnNotifyReviewerAsync(
        ILifeCycleHookEvent evt, ConsumerContext ctx) {

        // RunCount == 1: first time this hook has fired for this instance/state.
        // RunCount == 2: hook fired again after ReopenAsync — new ack_guid, new inbox row.
        // ACTION_SEND_EMAIL is scoped per-instance: each reopen gets its own instance row.

        if (ctx.RunCount > 1) {
            _log.LogWarning(
                "Hook re-run #{N} for instance {Guid} — prior notification may exist.",
                ctx.RunCount, ctx.InstanceGuid);
        }

        // ExecuteBusinessActionAsync skips the delegate if already Completed for this
        // instance_id + action_code. On re-delivery of the SAME dispatch (same ack_guid)
        // the action is also skipped. Both re-delivery and reopen cases are handled.
        await ExecuteBusinessActionAsync(ctx, STEP_SEND_EMAIL, async (ct) => {
            // Include RunCount in the subject so the reviewer knows it's a rerun
            var subject = ctx.RunCount > 1
                ? $"[Re-notification #{ctx.RunCount}] Loan {ctx.EntityGuid} needs review"
                : $"Loan {ctx.EntityGuid} needs your review";
            await _email.SendAsync(subject, ctx.EntityGuid, ct);
            return null;
        });

        return AckOutcome.Processed;
    }

    protected override Task<AckOutcome> OnUnhandledTransitionAsync(
        ILifeCycleTransitionEvent evt, ConsumerContext ctx)
        => Task.FromResult(AckOutcome.Processed);
    protected override Task<AckOutcome> OnUnhandledHookAsync(
        ILifeCycleHookEvent evt, ConsumerContext ctx)
        => Task.FromResult(AckOutcome.Processed);
}C#

What happens on ReopenAsync

StepEngine sideConsumer side
1. ReopenAsync calledForce-resets instance state; creates a new lifecycle row (new lc_id)—
2. PolicyEnforcer runsInserts fresh hook_lc rows (dispatched=0) for the new lc_id. Old hook_lc rows remain as history.—
3. Hook dispatchedRunCount = COUNT(hook_lc WHERE dispatched=1) = 2 (prior run = 1, this run = 1). New ack_guid issued.New inbox row created (new ack_guid, run_count=2). ctx.RunCount = 2. The instance row already exists — same instance_id.
4. Handler runs—ctx.RunCount > 1 → handler knows it's a rerun. Business actions for this instance are persistent — ExecuteBusinessActionAsync will skip already-completed actions from the prior run.
New ack_guid on every reopen
Each reopen creates a new ack_guid scoped to the new hook_lc row. The consumer inbox table gets a new row (deduplication key is UNIQUE(ack_guid)). The instance row is reused — same instance_id. Business actions from the prior run are still in business_action and will be found by ExecuteBusinessActionAsync on the rerun.
